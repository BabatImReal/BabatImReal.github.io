Fintech Data Engineering Project

Project Overview

A comprehensive end-to-end data engineering solution designed to process, transform, and deliver actionable insights from financial transaction data in real-time.

Data Ingestion

Built real-time data ingestion using Apache Kafka, streaming transactions from PostgreSQL (OLTP) database into the data lake for immediate processing.

Workflow Orchestration

Orchestrated complex data workflows with Apache Airflow and Dagster for automated scheduling, monitoring, and dependency management across the pipeline.

Data Transformation

Transformed raw data into analytics-ready models using dbt, applying modular SQL transformations, data quality tests, and business logic validation.

Cloud Data Warehouse

Integrated with Snowflake as the cloud data warehouse, optimizing performance through partitioning, clustering, and cost-efficient storage strategies.

ETL Automation

Implemented Python-based ETL scripts for data validation, schema mapping, and pipeline automation ensuring data integrity throughout the process.

Analytics & Reporting

Delivered a reliable data infrastructure supporting real-time dashboards, financial reporting, and advanced analytics for strategic decision-making.

Dimensional Modelling

  • Fact Table (FCT_ORDER_ITEM): Central fact table storing transactional order item data with metrics like quantity, price, shipping costs, and product dimensions.
  • Dimension Tables: Supporting dimension tables including DIM_ORDER_PAYMENT (payment details), DIM_SELLER (seller information), DIM_CUSTOMER (customer data), and DIM_PRODUCT (product details).
  • Dimensional Hierarchy: Product dimension includes category translation and name length attributes for multi-language support and analytics.
  • Order Review Dimension: Tracks customer reviews with review scores, comment messages, and timestamps for sentiment analysis.
  • Star Schema Design: Implemented a star schema with fact table at the center connected to dimension tables via foreign key relationships for optimized query performance.
  • Key Relationships: Established primary and foreign key relationships between fact and dimension tables ensuring data integrity and efficient joins.
  • Data Types and Constraints: Defined appropriate data types (VARCHAR, INT, DATETIME, DECIMAL) with proper constraints for each attribute to ensure data quality.

Data Transformation with dbt

Leveraging dbt (data build tool) to implement a medallion-style architecture, transforming raw batch data into analytics-ready datasets through progressive refinement layers.

Medallion Architecture Layers

Staging Layer (Bronze)

Purpose: Light transformations on raw source data
Operations: Type casting, column renaming, basic data cleaning
Models: stg__customers, stg__orders, stg__products, stg__sellers, stg__order_items, stg__order_payments, stg__order_reviews

Intermediate Layer (Silver)

Purpose: Business logic transformations
Operations: Data enrichment, joins, aggregations, business rules
Models: int__products (enriched with category translations), int__order_items (with calculated metrics)

Analytics Layer (Gold)

Purpose: Star schema implementation for analytics
Operations: Dimension & fact table creation, surrogate keys
Models: dim_customers, dim_products, dim_sellers, dim_order_payment, dim_order_reviews, dim_date, fact_orders

Transformation Flow

  • Source to Staging: Raw data from sc_raw_data tables flows through snapshots for CDC (Change Data Capture), then to staging models with consistent naming and type casting.
  • Staging to Intermediate: Staged data is enriched with business logic - products joined with category translations, order items with calculated shipping metrics.
  • Intermediate to Analytics: Final transformation into star schema with dimension tables (dim_*) surrounding the central fact_orders table for optimal query performance.
  • dbt Snapshots: Implemented SCD Type 2 snapshots for tracking historical changes in customers, products, sellers, orders, and order items.
  • Data Quality: dbt tests applied at each layer for uniqueness, not null constraints, referential integrity, and accepted value validations.

Data Orchestration

Implementing robust workflow orchestration to automate data pipeline execution, manage dependencies, and ensure reliable data delivery through scheduled and event-driven triggers.

Dagster Orchestration

Asset-Based Architecture

Dagster's software-defined assets allow declarative pipeline definition. Each data asset (tables, models) is defined with its dependencies, enabling automatic lineage tracking and materialization.

Batch Ingestion Job

Orchestrates CSV file loading from source to Snowflake. Handles batch_csv_files extraction and snowfl__batch_data staging, ensuring data lands in the raw layer before transformation.

dbt Transformation Job

Triggers dbt runs after successful ingestion. Executes snapshot refreshes, staging models, intermediate transformations, and analytics layer builds in proper dependency order.

Observability & Monitoring

Built-in asset lineage visualization, run history, and failure alerting. Dagster UI provides real-time pipeline status, asset materialization history, and data quality insights.

Avg. Review Score
