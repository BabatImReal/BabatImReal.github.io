Bike Sharing Data Analytics Project

Project Overview

End-to-end analytics project that transforms raw bike sharing data into interactive dashboards and a public-facing web application, providing deep insights into rider behavior, station performance, and the impact of weather across a full year of operations.

Usage & Operations Analysis

Analyzed 12 months of bike-sharing data with over 100,000 trip records and 4,000+ station status entries to uncover usage patterns, demand peaks, and operational bottlenecks across the system.

Data Engineering with Python & dbt

Processed and transformed raw data from trips, stations, weather, and station status into clean, well-modeled datasets using Python scripts and dbt models, enabling reliable downstream analytics.

Metabase Analytics Dashboard

Designed an interactive Metabase dashboard with 20+ visualizations to explore rider segments, trip patterns, station throughput, and how weather conditions influence ridership over time.

Public Analytics Web App

Built and deployed a public-facing analytics web application that embeds Metabase dashboards and narrative insights, allowing users to intuitively explore bike usage trends.

Motivation & Goals

This project started from a simple question: how can data make bike sharing more convenient for everyday riders? I wanted to go beyond static reports and explore richer, time-based data to understand how a city-wide network of bikes actually behaves throughout the day, across seasons, and under different weather conditions. A key goal is to surface this information in a way that helps users quickly find the nearest station with both available bikes and free racks, reducing uncertainty and making the system feel more reliable. By combining detailed operational data with an intuitive interface, the project aims to bridge the gap between raw infrastructure metrics and practical, rider‑centric insights.

Project Structure: Historical & Real-time

The project is split into two main parts. The first focuses on historical analysis using the SF Bay Area Bike Share dataset from Kaggle. This data is cleaned, modeled, and used to build analytical tables and BI visualizations that answer questions about long-term usage patterns, popular routes, and station performance over time.

The second part focuses on real-time analytics. Live station data is fetched from the public Nextbike API and ingested into NeonDB, where it powers real-time dashboards embedded in the bike sharing web application. The website exposes both a real-time BI dashboard and an interactive map, allowing users to see the current status of each station and quickly find nearby locations with available bikes and free racks.

Dataset: Nextbike API

The project is powered by data collected from the public Nextbike API, which exposes real-time information about bike sharing systems across multiple cities. The dataset combines live station status and availability (bikes, docks, coordinates), network metadata (cities, networks, station attributes), and time-based snapshots of usage into a single analytics layer. By regularly pulling and storing these API feeds, the project reconstructs a year of operations, enabling analysis of when and where bikes are checked out or returned, how station utilization evolves over time, and how external factors such as time of day, day of week, and weather conditions influence demand.

Data Workflow & Architecture

The architecture combines a batch analytics pipeline for historical data with a real-time pipeline for live station status. Both flows are designed to land data in well-modeled tables that power Metabase dashboards and the public web application.

Historical Batch Pipeline

Kaggle trip, station, and weather files are ingested and cleaned with Python, then transformed with dbt into dimensional tables (trips, stations, time, weather). These curated models feed Metabase dashboards and PDF reports that summarize long-term usage patterns and station performance.

Real-time Pipeline

The Nextbike API is polled on a schedule to capture live station status snapshots. These events are ingested into NeonDB, where they are stored in analytics-ready tables and exposed through Metabase and a Next.js frontend. The website reads directly from this real-time layer to render the live dashboard and interactive map of stations, bikes, and docks.

Dimensional Modeling for Analytics

To support fast and flexible analysis, the cleaned data is organized into a star schema with a central fact_station_status table and several dimensions. Each row in the fact table captures the status of a station at a specific point in time (bikes and docks available), while the dimensions provide rich context for slicing and filtering the metrics.

  • dim_station stores station attributes such as name, location (latitude/longitude), dock count, city, and installation date.
  • dim_time breaks the timeline into hour, day, month, year, and day of week to enable time-series and peak-period analysis.
  • dim_weather captures daily weather measures (temperature, humidity, wind, precipitation, events, etc.) that can be joined to station status to study weather impact on demand.
  • dim_trip links trips back to stations and time, enabling combined views of historical trip volume with live availability patterns.
